## Bug: Micro-Price Rescue Corruption

### Root Cause

`_maybe_rescue_visual_bruto` and `_maybe_rescue_visual_neto` in `postprocess.py` had a dual condition:

```python
if ratio >= 10 or ratio <= 0.1:
    return expected
```

The `ratio <= 0.1` path was **WRONG**: when `expected << bruto`, it means the OCR bruto is correct and `qty × price` is wrong (because qty is still truncated pre-anchor). Replacing correct OCR bruto with `qty × price` corrupted the value.

### Corruption Chain (step by step)

1. OCR reads truncated quantity (e.g., `1105882.353` instead of `1105882353`)
2. Price is micro (e.g., `0.0009` for Dolar Cable BONAR)
3. `expected = truncated_qty × micro_price` → tiny value (e.g., `995.29`)
4. `ratio = expected / bruto = 995 / 995000 = 0.001` → triggers `ratio <= 0.1`
5. OCR bruto (correct: `995000`) gets **replaced** with `expected` (wrong: `995.29`)
6. Later, quantity anchor builder restores the correct full quantity
7. Next rescue pass sees: `new_expected = correct_qty × micro_price = 995000` but bruto is now `995.29`
8. `ratio = 995000 / 995.29 ≈ 1000` → triggers `ratio >= 10` → replaces again
9. Final bruto = `correct_qty × micro_price = 995000` ← this is correct by coincidence
10. **But neto** follows the same chain and gets corrupted to `~1053529` through gastos miscalculation

### Fix

Remove the `ratio <= 0.1` condition from both functions. Only keep `ratio >= 10` (expected >> bruto), which handles the legitimate case where OCR bruto is too small.

## Lesson 1: Rescue Functions Must Be Asymmetric

When OCR produces a value and code computes an alternative (`qty × price`), the rescue logic must account for **which quantity** is being used at each stage:
- **Pre-anchor**: qty is truncated → `qty × price` is unreliable → do NOT replace OCR values
- **Post-anchor**: qty is corrected → `qty × price` is reliable → OK to replace

The `ratio <= 0.1` condition assumed that when `expected << actual`, the actual must be inflated. But for micro-prices with truncated quantities, `expected` is the one that's deflated.

## Lesson 2: Multi-Stage Pipelines Create Hidden Feedback Loops

The corruption was amplified because:
1. Rescue #1 (with wrong ratio) replaced correct bruto with tiny value
2. Anchor builder independently restored the correct quantity
3. Rescue #2 (post-anchor) now had correct qty but wrong bruto → overcorrected

**Takeaway**: When a pipeline has multiple correction stages, a wrong correction in an early stage can cascade and amplify through later stages.

## Lesson 3: OCR Variability ≠ Code Bug

8 boletos (12716, 12715, 27693, 27691, 34140, 44360, 45057, 51038) were expected in the web output but absent from our local pipeline. Investigation with `pdfplumber` confirmed these boletos **do not exist** in either source PDF. They were likely present in a different OCR run on the web (Datalab OCR is non-deterministic).

**Takeaway**: Always verify whether "missing" data is a code bug or a source data limitation before investigating the pipeline.

## Lesson 4: Frozen Inputs Must Match the Approved OCR Run

Root-level frozen files (`11688_STURMAN_JAVIER_Gallo_from_PDF.xlsx`) had 36 rows in Renta Fija Pesos (with duplicates and extra boletos from a different OCR run). The `_FULL_FLOW_11688/` fresh files had 18 rows. Using the wrong frozen inputs would produce a baseline that doesn't match the web's output.

**Takeaway**: When creating smoke tests, use frozen inputs from the SAME OCR run that produced the approved output.

## Lesson 5: Smoke Test Guard Rails

The Sturman 11688 smoke test includes three guards beyond cell-by-cell comparison:

1. **Inflation guard**: No quantity > 10B in non-aux sheets → catches 1000× OCR artifacts
2. **Micro-price guard**: Specific boletos must have exact neto values → catches rescue corruption
3. **AUX_SHEETS skip**: EspeciesVisual/EspeciesGallo/etc. are only dimension-checked, not cell-by-cell → avoids false positives from CUSIP codes (e.g., `33738E109` looks like scientific notation)

## Lesson 6: openpyxl `data_only=True` Pitfall

When saving a workbook generated by openpyxl (not Excel), formula cells have no cached value. Loading with `data_only=True` returns `None` for those cells, while the in-memory workbook has the formula string. This creates phantom diffs in smoke tests.

**Fix**: Load baseline with `data_only=False` (default) so both in-memory and loaded workbooks see the formula string.

## Lesson 7: pdfplumber for Independent Verification

When Datalab OCR results are questionable, `pdfplumber` provides a quick independent text extraction to verify what's actually in the PDF. Used this to confirm the 8 "missing" boletos were genuinely absent from the source PDFs.
